## 4. 阶段4手工联调用例
| 编号 | 用例 | 操作步骤 | 预期结果 |
|---|---|---|---|
| T-RP-01 | 居民创建报修 | 调用 `POST /api/repairs` 提交小区、标题、描述、联系方式 | 生成工单，状态为 `PENDING_ACCEPT` |
| T-RP-02 | 物业受理并分派 | 依次调用 `accept`、`assign` | 状态依次为 `ACCEPTED`、`ASSIGNED` |
| T-RP-03 | 维修员处理并提交 | 依次调用 `take`、`process`、`submit` | 状态进入 `WAIT_CONFIRM` |
| T-RP-04 | 居民确认解决 | 调用 `confirm` | 状态变为 `DONE` |
| T-RP-05 | 物业关闭工单 | 调用 `close` | 状态变为 `CLOSED` |
| T-RP-06 | 居民反馈未解决 | 在 `WAIT_CONFIRM` 或 `DONE` 状态调用 `reopen` | 状态变为 `REOPENED` |
| T-RP-07 | 非法流转拦截 | 对 `PENDING_ACCEPT` 直接调用 `submit` | 返回 `ILLEGAL_STATE_FLOW` |
| T-RP-08 | 越权拦截 | 非工单所属居民调用 `my/{id}` 或催单 | 返回 `FORBIDDEN` |

## 5. 并发与稳定性建议
- 活动报名并发：建议使用 JMeter/Gatling 做 200~500 并发压测，关注是否超卖。
- 报修接口稳定性：建议对 `assign/take/submit/reopen` 做并发回归，确认状态机无脏写。
- 建议增加 MockMvc 集成测试，覆盖统一异常响应结构与权限返回码。
